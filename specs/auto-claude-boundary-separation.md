# Auto-Claude Boundary Separation, TDD Intent, Pattern Conformance & Escalation

## Context

`auto_claude` (~/bin/auto_claude, ~2975 lines) is a reusable autonomous pipeline that drives Claude through plan → implement → test → review phases. `worktree_auto_claude` (~/bin/worktree_auto_claude, ~1282 lines) orchestrates parallel deliverables by forking git worktrees and running independent `auto_claude` instances.

Five problems exist:

1. **Parallel run bleed** — When `worktree_auto_claude` forks deliverables whose file boundaries aren't cleanly separated, parallel Claude sessions independently implement the same helpers, refactor the same shared files, or make conflicting assumptions about shared types. The only current boundary mechanism is the `PLAN_CONTEXT` string in each worktree's `.auto_claude.conf`, which is advisory — Claude interprets it, no filesystem enforcement exists. Cherry-pick conflicts are the only detection mechanism and they fire too late.

2. **Context drift on iteration** — In `phase_test_fix`, each iteration sends identical prompt structure ("Tests are failing... fix the failures") relying on `--continue` conversation memory. By iteration 3+, Claude's own failed fixes dominate the context window, pushing the original spec intent and decisions out of effective attention. There is zero structured tracking of what approaches were tried or why they failed.

3. **No fix loop escape hatch** — The test_fix loop is a flat retry: try up to `MAX_TEST_ITERATIONS` (default 5), then hard fail. When the same class of failure repeats, Claude is given the same framing each time — no mechanism to step back, reconsider the approach, consult broader architecture, or involve the human with structured alternatives.

4. **Intent is not machine-checkable** — The intent lives in the spec (natural language) and the decisions manifest (structured but still prose). When tests fail, the gap between "what was intended" and "what happened" is buried in 400 lines of test output. There is no structured mapping from spec requirements to test assertions, so escalation can't tell you *which intent* failed or *why*.

5. **Pattern conformance is implicit** — Claude optimises for the prompt's local goal, not consistency with surrounding code. The `OPINIONATED` flag says "prefer simpler, defer to codebase patterns" but nobody checks whether the implementation actually conforms. Novel patterns get introduced silently — sometimes justified, often not — and the review phases don't distinguish between intentional deviations and accidental drift.

This spec is designed to run via `~/bin/auto_claude`.

### What this spec does NOT do

- Does not change the authoring/quality seam boundary (`SEAM_AFTER="ci_checks"` stays)
- Does not modify `run_claude` / `run_claude_fresh` / `_invoke_claude` calling conventions
- Does not change cherry-pick or worktree lifecycle logic
- Does not add new external dependencies (stays bash + jq + git)
- Does not change `phase_plan` or `phase_decisions` behaviour (only adds phases between/after them)
- Does not change the final auto-commit logic

### Dependencies (all met)

- [x] `auto_claude` exists at `~/bin/auto_claude`
- [x] `worktree_auto_claude` exists at `~/bin/worktree_auto_claude`
- [x] jq available (already a hard dependency)

- [x] `_build_context_pack` case switch supports adding new pack variants

### Glossary

- **Ownership manifest** — A JSON file mapping each parallel run to the files it owns (read-write), may read (read-only), and must not touch. Generated before worktrees fork.
- **Approach journal** — A structured log of what was tried in each test_fix iteration: one-line summary, files touched, error class. Persisted in the state file's `deterministic` section.
- **Tiered escalation** — Replacing the flat retry loop with distinct strategies per iteration tier: tactical fix → re-anchored fix → architectural consultation → human checkpoint.
- **Boundary fence** — A git pre-commit hook written into each worktree that rejects changes to files in the `must_not_touch` list.
- **Intent map** — A structured mapping from spec requirements to test assertions, generated by `phase_test_intent`. Each entry links a test name to the expected behaviour it encodes.
- **TDD layers** — Test types ordered by cost: unit → integration → component → e2e. Configurable via `TDD_LAYERS` to control which layers are written in `phase_test_intent`.
- **Pattern baseline** — A snapshot of coding patterns in the files adjacent to the change area, derived fresh each pipeline run. Not persisted beyond the run.
- **Pattern deviation** — An implementation choice that differs from the derived baseline. Each deviation gets a structured pros/cons analysis and a verdict: revert (conform to existing) or keep (with a `PATTERN-DEVIATION` comment).
- **Three-tier pattern model** — Tier 1: deliberate architectural decisions (centralised in CLAUDE.md). Tier 2: codebase patterns (derived fresh, ephemeral). Tier 3: deviation decisions (distributed as comments near the code).

## Deliverable 1: Ownership Manifest (worktree_auto_claude)

**Files:**
- `~/bin/worktree_auto_claude` (modify)
- `~/bin/auto_claude` (modify — read manifest in plan/implement phases)

### Why

Parallel runs currently share the same codebase snapshot with no file-level constraints. `PLAN_CONTEXT` tells Claude which deliverables to implement but doesn't prevent it from touching shared files. When two parallel sessions independently add a utility function to the same barrel re-export or refactor the same type definition, the cherry-pick merge fails — or worse, silently produces duplicated code if the changes don't conflict textually.

The ownership manifest makes boundaries enforceable: each worktree gets a pre-commit hook that hard-blocks changes to other runs' files, and the `PLAN_CONTEXT` is enriched with explicit read-only constraints.

## Deliverable 2: Approach Journal (auto_claude)

**File:** `~/bin/auto_claude` (modify)

### Why

The test_fix loop currently sends identical prompt structure each iteration, relying entirely on Claude's `--continue` conversation memory to track what was tried. This has two failure modes:
- By iteration 3+, Claude's own failed fix attempts dominate the attention window, pushing original intent out of effective recall
- There's no structured record of what was tried, so Claude can (and does) attempt the exact same fix twice

An approach journal tracks each iteration's strategy in the state file and re-injects it into subsequent prompts. It's cheap (one extraction call per iteration, ~50 tokens stored per entry) and directly prevents repeated attempts.

## Deliverable 3: Tiered Escalation (auto_claude)

**File:** `~/bin/auto_claude` (modify)

### Why

The flat test_fix loop repeats the same strategy (send test output → ask Claude to fix) regardless of iteration count. When the failure is a design-level mistake — wrong data model, incorrect adapter interface, flawed state management approach — tactical code fixes don't converge. Claude gets deeper into its own failing approach instead of stepping back.

Tiered escalation replaces the flat loop with distinct strategies per tier:
- **Tier 1 (iterations 1-2):** Tactical fix — same as current behavior, plus approach journal. Most failures are caught here.
- **Tier 2 (iteration 3):** Re-anchored fix — re-inject spec intent + decisions manifest + full approach journal. Forces Claude to check whether its approach aligns with the original design.
- **Tier 3 (iteration 4):** Architectural consultation — fresh session with broader context (surrounding modules, dependency graph, approach journal, intent map from D4 if available). Claude proposes 2-3 alternative approaches with tradeoffs. Saved to state and explain dir.
- **Tier 4 (iteration 5):** Human checkpoint — present the alternatives from tier 3 with structured intent-vs-reality gap analysis, pause for input. Human picks an approach (or aborts), pipeline resumes with a focused re-implementation.

## Deliverable 4: Test-Intent Phase (auto_claude)

**File:** `~/bin/auto_claude` (modify)

### Why

The current pipeline writes tests alongside implementation in `phase_implement`. This means:
- Tests are shaped by the implementation, not the spec — they test what was built, not what was intended
- When tests fail in test_fix, the failure output is raw and unstructured — you can't tell which *intent* failed without reading the test code
- The escalation system (D3) has no structured intent data to build gap analyses from — it works with test names and output, not intent descriptions

TDD inverts this: write tests first from the spec, then implement to make them pass. The tests become machine-readable intent — each test name and assertion encodes a specific acceptance criterion. When implementation fails, the gap between "what the test expects" and "what the code does" IS the structured understanding of the problem.

## Deliverable 5: Pattern Conformance (auto_claude)

**File:** `~/bin/auto_claude` (modify)

### Why

Claude optimises for the prompt's local goal — implementing the spec — not for consistency with code it hasn't been told to care about. The `OPINIONATED` flag gestures at "prefer codebase patterns" but it's a blunt, unverifiable instruction. Nobody checks whether the implementation actually conforms to surrounding patterns. Novel patterns get introduced silently — sometimes justified (the existing pattern genuinely can't accommodate the requirement), often not (Claude defaulted to its training distribution instead of the codebase's convention).

Pattern conformance makes this checkable: derive the patterns from adjacent code, compare the implementation against them, produce structured pros/cons for each deviation, and act on the verdict — revert to existing if cons outweigh, keep with a documented `PATTERN-DEVIATION` comment if pros outweigh.

### Three-tier pattern model

The system uses three tiers of pattern authority, each with a different storage strategy:

**Tier 1 — Deliberate architectural decisions (centralised in CLAUDE.md)**
Already exists. "Discriminated unions over optional fields." "All data flows through hooks.ts." "Files under 300 LOC." These are prescriptive — they describe what code *should* do. They survive refactors because they're intent, not observation. No changes needed to this tier.

**Tier 2 — Codebase patterns (derived fresh each run, ephemeral)**
This is what `phase_pattern_baseline` produces. "All 3 adapters use try/catch with typed errors." "All 8 hooks call useDataSource()." Derived by reading the actual code adjacent to the change area. Not stored permanently — regenerated each run, so it's always current. Lives in the state file for the duration of the pipeline.

**Tier 3 — Deviation decisions (distributed as comments near the code)**
When the conformance review decides to keep a deviation, it adds a `PATTERN-DEVIATION` comment at the deviation site:
```typescript
// PATTERN-DEVIATION: Uses local useState instead of React Query here because
// disambiguation results are ephemeral UI state, not server data that needs
// caching or refetching. See DEV-004.
```
Auto_claude recognises these markers on future runs and doesn't re-litigate the decision — unless the surrounding context has changed enough that the rationale no longer holds.
